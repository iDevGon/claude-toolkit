# Smart Commit & Push

git 변경사항을 분석하여 논리적 단위로 나누어 커밋하고, 리모트에 푸쉬하는 스킬.

## 실행 절차

### 1단계: 현재 상태 파악

아래 명령을 **병렬로** 실행하여 현재 상태를 파악한다.

```bash
# 변경된 파일 목록 확인
git status

# staged + unstaged 변경사항 확인
git diff HEAD

# 최근 커밋 메시지 스타일 확인 (최근 20개)
git log --oneline -20

# 현재 브랜치와 리모트 tracking 상태 확인
git branch -vv
```

### 2단계: 변경사항 분석 및 그룹핑

수집한 정보를 바탕으로 변경사항을 논리적 단위로 분류한다.

**분류 기준** (우선순위 순):

1. **기능 단위**: 하나의 기능 추가/변경에 관련된 파일들을 하나의 커밋으로 묶는다.
2. **레이어 단위**: 같은 레이어(예: API, 서비스, 모델, 테스트)의 관련 변경을 묶는다.
3. **의존성 순서**: 다른 변경에 의존하는 변경이 있다면, 의존되는 변경을 먼저 커밋한다.
4. **설정/인프라 변경**: 설정 파일, 의존성 변경 등은 별도 커밋으로 분리한다.
5. **리팩토링 분리**: 기능 변경과 리팩토링은 별도 커밋으로 분리한다.

**분류 시 주의사항**:

- 하나의 파일이 여러 논리적 변경을 포함할 수 있다. 이 경우 `git add -p`를 사용하여 hunk 단위로 staging한다.
- 너무 잘게 나누지 않는다. 밀접하게 관련된 변경은 하나의 커밋으로 유지한다.
- 변경사항이 모두 하나의 논리적 단위라면 굳이 나누지 않고 하나의 커밋으로 만든다.

### 3단계: 커밋 메시지 작성

**스타일 규칙**:

- 1단계에서 확인한 기존 커밋 메시지의 문체와 스타일을 따른다.
- 사용자가 별도 지시를 하면 그 지시를 우선한다.
- 기존 히스토리가 없거나 스타일을 판단하기 어려운 경우, Conventional Commits 형식을 기본으로 사용한다:
  - `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`, `style:`, `ci:` 등의 prefix
  - 첫 줄은 50자 이내 권장, 필요시 본문에 상세 설명 추가

**메시지 내용 규칙**:

- "what"보다 "why"에 초점을 맞춘다.
- 변경의 목적과 의도를 명확히 전달한다.

### 4단계: 커밋 실행

각 논리적 단위별로 순서대로 커밋한다.

```bash
# 1) 관련 파일만 staging (특정 파일)
git add <file1> <file2> ...

# 또는 hunk 단위 staging이 필요한 경우
# git add -p <file>  (사용자에게 안내 후 직접 실행하도록 유도)

# 2) 커밋 생성 (HEREDOC 사용)
git commit -m "$(cat <<'EOF'
커밋 메시지 제목

필요시 본문 설명

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
EOF
)"
```

**주의사항**:

- `git add -A`나 `git add .`는 사용하지 않는다. 반드시 파일을 명시적으로 지정한다.
- `.env`, `credentials.json` 등 민감한 파일은 절대 커밋하지 않는다. 발견 시 사용자에게 경고한다.
- 커밋 후 `git status`로 결과를 확인한다.
- `--no-verify`, `--amend` 등은 사용자가 명시적으로 요청한 경우에만 사용한다.
- hunk 단위 staging(`git add -p`)이 필요한 경우, interactive 입력이 필요하므로 사용자에게 `! git add -p <file>` 명령을 안내하여 직접 실행하도록 한다.

### 5단계: 리모트 푸쉬

모든 커밋이 완료되면 리모트에 푸쉬한다.

```bash
# upstream tracking이 설정되어 있는 경우
git push

# upstream tracking이 없는 경우 (새 브랜치)
git push -u origin <현재-브랜치명>
```

**푸쉬 주의사항**:

- `--force` 또는 `--force-with-lease`는 사용자가 명시적으로 요청한 경우에만 사용한다.
- main/master 브랜치에 직접 푸쉬하는 경우, 사용자에게 확인을 요청한다.
- 푸쉬 실패 시(예: reject) 원인을 분석하여 사용자에게 안내한다. 임의로 force push하지 않는다.

### 6단계: 결과 보고

모든 작업 완료 후, 생성된 커밋 목록과 푸쉬 결과를 간결하게 보여준다.

```bash
git log --oneline -<생성한 커밋 수>
```
